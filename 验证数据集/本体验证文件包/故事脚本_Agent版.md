# 故事脚本 · manus 式自主智能体（本体 + 图 + Agent）

> 工具调用 = function-calling（用 `tools.json`）；结果按 **JSON** 回灌；Agent **自主**决定调谁/几次/顺序/何时停——不预设流程。
> 用你自己的 LLM（OpenAI 兼容 function-calling 接口）跑。运行说明见 `运行说明_Agent.md`。
>
> **判定原则**：每个故事都是 workflow 跑不完的"调查任务"——目标给定、路径不给。价值在于 Agent 自主规划 + 在本体边界内 + 用图查关系 + 多跳假设/推翻/再验证 + 出有论证的判断。

## 三者如何协同（每个故事都体现）
- **本体**：先 `ontology_get_slice` 取切片→Agent 知道能动哪些字段、术语统一、越界不可读（边界/世界模型）。
- **图**：`graph_traverse/common_neighbors/shortest_path`→查关系、穿透、拼隐藏链路（关系底座）。
- **Agent(manus式)**：目标驱动、自主编排上述工具、组合弱信号、形成判断并自评（调查员）。

## 故事一 · 隐藏关联网络的自主侦查 〔本体边界 + 图穿透 + Agent 自主规划〕

**任务（只给目标）**：评估即将与 **海岳贸易 S-301** 签订的 250 万合同 `C-2024-007` 的对手方风险，给出是否可签的判断与依据。

**为什么 workflow 不行**：固定流程只查 S-301 自身→干净（征信78、不在黑名单、条款齐全）→放行。风险藏在关系与弱信号里。

**期望的 Agent 自主轨迹（多轮 function-calling，JSON 回灌）**：
1. `ontology_get_slice("subject-due-diligence")` → 知道可查 UBO/关联/外部事实，且不可越界看付款明细。
2. `entity_get("S-301")` → 字段干净。Agent 自评："表面无问题，但需查实际控制人，不能只看自身。"（自主决定深挖）
3. `graph_traverse(startId="S-301", rel="HAS_SHAREHOLDER", hops=3)` → 返回路径 `S-301→海岳控股→王某(UBO)`。
4. Agent 起新假设："王某还控制谁？" → `graph_traverse(startId="王某", rel="HAS_SHAREHOLDER", hops=2, direction="in")` → 命中 `S-900(黑名单)` 与 `S-204(曾被关注)`。
5. 形成假设："疑似借壳重投" → 主动取证拼图：`graph_common_neighbors("S-301","S-204")` + `external_lookup("S-301","registry")` + `external_lookup("S-301","bankflow")` → 拿到**同楼相邻地址、同一代办(粤信财税)、相近银行账户、S-301向S-204转账48万**。
6. `rule_evaluate("C-2024-007")` → 符号规则只报"无硬否决"（印证 workflow 会放行），Agent 据此知道"这案子规则救不了，要靠关系判断"。
7. **结论**：高风险——"S-301 与已被关注的 S-204 共享最终受益人王某，且王某控制黑名单实体 S-900；叠加同楼地址/同代办/相近账户/资金往来等多条弱信号，研判为**疑似借壳重新投标**。建议暂缓签署，启动深度尽调。" + **证据 id 列表** + **不确定性自评**（弱信号为间接证据，建议人工核验银行流水）。

**断言**：A1 自身干净仍判高风险；A2 给出完整穿透+弱信号证据链；A3 与纯规则(放行)对照；**A4 Agent 自主发起了未被预设的深挖步骤（横向扩展+拼图），而非固定流程**。

## 故事二 · 条款语义对抗与跨条款交叉影响 〔本体 + 非结构化语义 + Agent 综合判断〕

**任务**：评估 `C-2024-001` 对我方的**真实风险敞口**，给出修订优先级。

**为什么 workflow 不行**：clauseType 标签齐全(付款/质量/责任上限/保密)，硬规则全过；真实风险藏在**条款原文语义**和**跨条款组合**里。

**期望的 Agent 自主轨迹**：
1. `ontology_get_slice("legal-procurement")` → 明确职责边界。
2. `rule_evaluate("C-2024-001")` → 硬地基：缺"数据合规"强制条款(确定性)；其余无否决。Agent："规则只发现缺条款，但要读原文看有没有被架空的风险。"
3. `clause_read_text("C-2024-001")` → 读到原文，**语义研判**发现规则看不出的三处：
   - 责任上限条款写了"以合同金额为限"但**被"仅限直接损失"暗中架空**（间接/利润/数据损失全免）；
   - 付款"先付90%、验收在最后"与交付里程碑**严重错配**（钱付了货没保障）；
   - 保密条款"至甲方认为无需保密时止"=**开放式无期限陷阱**。
4. 起假设"主体履约弱会放大上述敞口" → `external_lookup("S-204","litigation")` + `entity_get("S-204",["paidCapital","establishedDate"])` + 读历史合作 → 实缴50万/成立<12月/涉诉3起/历史2次延期。
5. **跨条款交叉推理**：付款错配 × 履约弱 × 责任免间接损失 → "若延期或质量问题，我方既已付90%又难追偿间接损失，敞口被多重放大。"
6. **结论**：真实敞口远高于规则所示；给**优先级修订建议**（① 改责任上限删除"仅限直接损失" ② 付款改为到货/验收后为主 ③ 保密设明确期限 ④ 补数据合规条款），每条引用条款 id 与事实 id。
7. 审核员追问"为什么先改责任上限？"→ Agent 引证据解释（S7 问答，反幻觉）。

**断言**：B1 多源综合敞口判断；B2 单规则不可得；**B3 语义发现——读原文挑出被架空的责任上限/错配/无期限**；B4 跨条款交叉推理；B5 可解释问答引用真实 id。

## 故事三 · 政策变更影响面评估与差异化处置 〔本体一致性 + Agent 判断性处置〕

**任务**：个保法收紧、政策 v2 新增"数据合规"强制项。评估在手合同暴露，并给出**每份合同的差异化处置方案**。

**为什么 workflow 不行**："谁缺这条"是查询(workflow 可做)；但**每份合同该怎么处置**是判断——要综合 合同状态/金额/是否涉个人信息/对方配合度/已履约进度。

**期望的 Agent 自主轨迹**：
1. Agent 自己规划："先找暴露面，再逐个定处置。"
2. 对每份在手合同 `rule_evaluate(contractId, policyVersion="v2")` → 找出缺"数据合规"的：001/005/006（007 已含、088 在 v1 下当时合规）。
3. 对每份 `entity_get(contract, ["status","amount","involvesPII","counterpartyCoop","fulfillmentProgress"])` → 取处置所需上下文。
4. **差异化判断**（这是 Agent 价值，非查询）：
   - `C-2024-001`（审核中、120万、涉PII、已付预付、对方配合一般）→ **签前必补数据合规条款，且因已付预付，建议同时加担保**；
   - `C-2024-005`（审核中、不涉PII、对方配合高）→ 风险低，**补充条款即可，快速通道**；
   - `C-2024-006`（审核中、涉PII、对方配合低）→ **暂缓生效，发函要求补充，配合度低需升级审批**；
   - `C-2023-088`（已生效、已履约完成、涉PII）→ **存量合同，评估残余风险；履约已完成则签补充协议留痕，不影响已完成交易**。
5. 用 v1 重放 088 证明"当时通过是合规的"（可复现/免责），用 v2 重放证明差异可溯。
6. **结论**：暴露清单 + 每份的处置策略与理由 + 优先级 + 工作量预估。

**断言**：C1 改政策一处全局一致命中；C2 历史 v1/v2 重放可溯；**C3 差异化处置——同样"缺条款"，Agent 据上下文给出不同处置而非统一动作**；C4 处置理由可解释。
